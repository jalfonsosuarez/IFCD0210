¿Qué son las normas ARIA?

ARIA (Accessible Rich Internet Applications) es un conjunto de especificaciones del W3C cuyo objetivo es mejorar la accesibilidad de aplicaciones web dinámicas,UF1843: Aplicaciones técnicas de usabilidad y accesibilidad en el entorno cliente especialmente aquellas construidas con JavaScript, cuando HTML semántico no es suficiente.

UF1843: Aplicaciones técnicas de usabilidad y accesibilidad en el entorno cliente ARIA no sustituye a HTML, lo complementa.

1. ¿Por qué existen las normas ARIA?

HTML por sí solo no siempre puede describir correctamente:

  • Componentes dinámicos (modales, pestañas, acordeones)
  • Cambios de contenido sin recarga de página
  • Estados complejos de la interfaz
  • Aplicaciones tipo SPA

ARIA permite comunicar esa información a las tecnologías de asistencia (por ejemplo, lectores de pantalla).

2. ¿Qué definen las normas ARIA?

ARIA define tres tipos principales de atributos:

2.1 Roles

Indican qué es un elemento.

  • role="navigation"
  • role="button"
  • role="dialog"
  • role="alert"
<div role="button">Guardar</div>

2.2 Estados (states)

Indican en qué estado está un elemento.

  • aria-expanded
  • aria-checked
  • aria-disabled
  • aria-hidden
<button aria-expanded="false">Menú</button>

2.3 Propiedades (properties)

Aportan información adicional.

  • aria-label
  • aria-labelledby
  • aria-describedby
  • aria-live
<input aria-describedby="help-text">

3. Ejemplos prácticos de uso correcto

3.1 Modal accesible

<div role="dialog" aria-modal="true" aria-labelledby="titulo-modal">
  <h2 id="titulo-modal">Confirmación</h2>
</div>

El lector de pantalla entiende que es un diálogo modal.

3.2 Botón con estado expandido

<button aria-expanded="true">
  Mostrar filtros
</button>

El usuario sabe si el contenido está abierto o cerrado.

3.3 Contenido dinámico anunciado

<div aria-live="polite">
  Resultados actualizados
</div>

El cambio se anuncia sin interrumpir al usuario.

4. Reglas fundamentales de ARIA

Regla 1 — Usa HTML nativo primero

Si existe un elemento HTML para una funcionalidad, no uses ARIA.

Incorrecto:

<div role="button">Enviar</div>

Correcto:

<button>Enviar</button>

Regla 2 — No arregles HTML roto con ARIA

ARIA no:

  • Añade foco de teclado
  • Corrige contraste
  • Arregla el orden del DOM

Regla 3 — No uses ARIA si no sabes exactamente por qué

ARIA mal usada empeora la accesibilidad.

Regla 4 — El estado ARIA debe reflejar la realidad

<button aria-expanded="false">

Si el menú está abierto, este valor es un error grave.

5. Errores comunes con ARIA

Error Impacto
aria-label innecesario Oculta texto visible útil
Roles duplicados Confusión en lectores de pantalla
aria-hidden="true" mal usado Contenido invisible para usuarios
ARIA sin soporte de teclado Interacción inaccesible
Copiar/pegar ARIA Errores graves difíciles de detectar

6. ARIA y WCAG

  • ARIA no es un criterio WCAG
  • Su mal uso puede causar incumplimiento
  • WCAG exige:
    • Nombre accesible
    • Rol correcto
    • Estado coherente

7. ARIA en frameworks modernos

React / Vue / Angular

  • Necesaria en SPAs
  • Riesgo alto si se usa mal
  • Preferir componentes accesibles probados

Plone / Volto

  • ARIA integrada en componentes core
  • Buena base accesible por defecto
  • El riesgo aparece en desarrollos personalizados

8. Resumen técnico

ARIA es una herramienta quirúrgica:

  • Útil cuando se necesita
  • Peligrosa si se abusa

Principios finales:

  • HTML primero
  • ARIA solo cuando hace falta
  • Probar siempre con lector de pantalla
